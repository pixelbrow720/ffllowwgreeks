### Structural observations

1. **Two-tier doc structure works.** Workspace `docs/` for cross-cutting concerns + `backend/docs/` for backend-specific deep dives + `backend/docs/reference/` for the validated subsystem refs. Each layer has a clear role.
2. **`backend/docs/reference/` is exemplary.** Validated against a specific commit, file:line citations, has an explicit "what's deliberately not here" section. Other doc folders could borrow that header pattern.
3. **Source-of-truth stack is well-defined** in `backend/HANDOFF.md`: HANDOFF > reference > PROGRESS > REVIEW > CHANGELOG > git log. Workspace HANDOFF declares its own ordering. Both rankings agree where they overlap. Good.
4. **Append-only docs (PROGRESS.md, CHANGELOG.md, REVIEW.md) accumulate stale content over time.** Annotation-in-place ("[superseded by auth pivot]") is the cheapest way to keep them honest without violating their append-only contract. Recommended over rewriting.
5. **Single biggest cleanup win:** resolve the `design-reference/` orphan. Five files reference it as a live folder; if it's genuinely missing, every reference becomes a broken link. If it exists and the audit's `ls` was misled by a Windows quirk, no action needed — but verify.
6. **No empty placeholder files** found in scope.
7. **No file links to a missing `.md`** other than the `docs/README.md` "planned" stubs noted above.
